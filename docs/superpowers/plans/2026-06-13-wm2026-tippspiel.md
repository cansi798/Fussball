# WM 2026 Tippspiel – Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans to implement task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Eine responsive Webapp, in der Kinder & Eltern WM-2026-Ergebnisse tippen, getrennt nach Vereinen, mit Punktewertung, Ranglisten, Spielplan (inkl. dt. TV-Sender) und privatem Admin-Bereich.

**Architecture:** Statisches React-Frontend (GitHub Pages) spricht ausschließlich mit Supabase. Kein E-Mail-Auth: Registrierung über Vereins-Einladungscode, Login über Benutzername+PIN. Eine Edge Function signiert ein eigenes JWT mit Supabase-Claims (`verein_id`, `teilnehmer_id`, `rolle`, `haushalt`); damit greifen die RLS-Policies in Postgres. Ergebnisse kommen on-demand aus OpenLigaDB (keyless), Punkte rechnet ein DB-Trigger.

**Tech Stack:** React 18 + Vite + TypeScript + TailwindCSS + react-router; @supabase/supabase-js; Supabase (Postgres, RLS, Deno Edge Functions); Vitest; GitHub Actions.

## Phase 1 — Datenbank

### Task 1: Schema (`supabase/migrations/0001_schema.sql`)
- [ ] Enums: `rolle` (`kind`,`elternteil`,`admin`), `phase` (`gruppe`,`ko`), `sieger` (`heim`,`gast`).
- [ ] Tabellen `verein, teilnehmer, benutzer, mannschaft, spiel, tipp, verfolgt` exakt nach Spec §4.
- [ ] `verfolgt`: Trigger/Constraint „max. 3 pro Kind".
- [ ] `tipp`: UNIQUE(teilnehmer_id, spiel_id).
- [ ] Indizes auf FKs + `spiel.anstoss`.

### Task 2: Scoring (`0002_scoring.sql`)
Reine Funktion `punkte_fuer(tipp_h,tipp_g,real_h,real_g,phase,elfmeter_sieger)`:
- exakt → 3; genau eine Torzahl → 1; sonst 0.
- K.o.+Elfmeter: +1 wenn Tipp Nicht-Unentschieden zugunsten des Siegers.
Trigger auf `spiel` (AFTER UPDATE von Toren/Status): alle `tipp` des Spiels neu berechnen.

### Task 3: RLS (`0003_rls.sql`)
- Claims aus `auth.jwt()`: `verein_id`, `teilnehmer_id`, `rolle`, `haushalt`.
- `teilnehmer`/`tipp`/`verfolgt`: SELECT nur eigener Verein; `admin` alles.
- `tipp` INSERT/UPDATE WITH CHECK: Ziel-Teilnehmer = self ODER `kind` im selben Haushalt, UND `(SELECT anstoss FROM spiel WHERE id=spiel_id) > now()`.
- Fremde Tipps erst nach Anstoß sichtbar.

### Task 4: Views (`0004_views.sql`)
- `rangliste`: pro teilnehmer Summe punkte, #exakt, #teiltreffer, #tipps; sortierbar.
- Frontend filtert nach Rolle (Gesamt/Kinder/Eltern) + Verein.

## Entscheidungen (autonom getroffen)
- **Auth:** Custom HS256-JWT via Edge Function (Supabase-JWT-Secret), Claims steuern RLS. Frontend nutzt Token als `Authorization: Bearer` für PostgREST.
- **Hashing:** bcrypt für PIN und Einladungscode.
- **Teams/Spiele:** dynamisch via Sync (nicht hartkodiert), da von Qualifikation/Auslosung abhängig.
- **TV:** Default „MagentaTV" (zeigt alle Spiele) für jedes Spiel; ARD/ZDF pro Spiel im Admin überschreibbar.
- **Repo-Base:** GitHub Pages unter `/Fussball/` (laut `Github.txt`).
